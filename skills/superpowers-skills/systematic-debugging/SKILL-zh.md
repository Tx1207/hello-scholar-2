# 系统化调试

## 概述

随机修复会浪费时间并制造新 bug。快速补丁会掩盖底层问题。

**核心原则：** 在尝试修复前，始终先找到根因。修症状就是失败。

**违背这个流程的字面规则，就是违背调试精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果你还没有完成第 1 阶段，就不能提出修复。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**在以下情况尤其要使用：**
- 有时间压力（紧急情况会诱使人猜测）
- “就快速修一下”看起来很明显
- 你已经尝试过多个修复
- 上一个修复没有奏效
- 你没有完全理解问题

**不要在以下情况跳过：**
- 问题看起来简单（简单 bug 也有根因）
- 你很赶时间（仓促会保证返工）
- 经理希望现在就修好（系统化比乱试更快）

## 四个阶段

你必须完成每个阶段后才能进入下一个阶段。

### 第 1 阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们经常包含准确的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠触发它吗？
   - 精确步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要猜

3. **检查近期变更**
   - 哪些变化可能导致这个问题？
   - Git diff、近期提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → build → signing，API → service → database）：**

   **在提出修复前，添加诊断埋点：**
   ```
   For EACH component boundary:
     - Log what data enters component
     - Log what data exits component
     - Verify environment/config propagation
     - Check state at each layer

   Run once to gather evidence showing WHERE it breaks
   THEN analyze evidence to identify failing component
   THEN investigate that specific component
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这会揭示：** 哪一层失败（secrets → workflow ✓，workflow → build ✗）

5. **追踪数据流**

   **当错误位于很深的调用栈中时：**

   完整的反向追踪技术见本目录中的 `root-cause-tracing.md`。

   **快速版：**
   - 错误值从哪里产生？
   - 是什么用错误值调用了这里？
   - 持续向上追踪，直到找到源头
   - 在源头修复，而不是在症状处修复

### 第 2 阶段：模式分析

**先找到模式，再修复：**

1. **寻找可工作的示例**
   - 在同一代码库中找到类似且可工作的代码
   - 与坏掉部分相似的可工作内容是什么？

2. **与参考实现对比**
   - 如果在实现某种模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 完全理解模式后再应用

3. **识别差异**
   - 可工作和坏掉的部分有什么不同？
   - 列出每一个差异，无论多小
   - 不要假设“那个不可能有影响”

4. **理解依赖**
   - 它需要哪些其他组件？
   - 需要哪些设置、配置、环境？
   - 它依赖哪些假设？

### 第 3 阶段：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚陈述：“我认为 X 是根因，因为 Y”
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做尽可能小的变更来测试假设
   - 一次只改变一个变量
   - 不要一次修多个东西

3. **先验证再继续**
   - 有效？是 → 第 4 阶段
   - 无效？形成新假设
   - 不要在上面继续叠加修复

4. **当你不知道时**
   - 说“我不理解 X”
   - 不要假装知道
   - 寻求帮助
   - 继续研究

### 第 4 阶段：实现

**修复根因，而不是症状：**

1. **创建失败测试用例**
   - 最简单的复现
   - 如有可能，写自动化测试
   - 没有测试框架时，写一次性测试脚本
   - 修复前必须有
   - 使用 `superpowers:test-driven-development` 技能编写合适的失败测试

2. **实现单一修复**
   - 处理已识别的根因
   - 一次只做一个变更
   - 不做“既然来了顺手改”的改进
   - 不捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试吗？
   - 问题确实解决了吗？

4. **如果修复无效**
   - 停止
   - 计数：你已经尝试了多少次修复？
   - 如果 < 3：回到第 1 阶段，带着新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（见下面第 5 步）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **如果 3 次以上修复都失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复都会在不同位置暴露新的共享状态/耦合/问题
   - 修复需要“巨大重构”才能实现
   - 每次修复都会在别处制造新症状

   **停止并质疑基本前提：**
   - 这种模式从根本上合理吗？
   - 我们是否“只是因为惯性才坚持它”？
   - 应该重构架构，而不是继续修症状吗？

   **在尝试更多修复前，与你的人类伙伴讨论**

   这不是假设失败——这是架构错误。

## 危险信号 - 停止并遵循流程

如果你发现自己在想：
- “现在快速修一下，之后再调查”
- “先试着改 X 看看是否有效”
- “添加多个变更，然后跑测试”
- “跳过测试，我会手动验证”
- “很可能是 X，我来修它”
- “我没完全理解，但这样可能有效”
- “模式显示 X，但我会用不同方式适配它”
- “主要问题如下：[列出修复但没有调查]”
- 在追踪数据流前提出解决方案
- **“再试一次修复”（当已经尝试 2 次以上）**
- **每次修复都在不同位置暴露新问题**

**所有这些都意味着：停止。回到第 1 阶段。**

**如果 3 次以上修复失败：** 质疑架构（见第 4.5 阶段）

## 你的人类伙伴发出的“你做错了”信号

**注意这些重定向：**
- “这不是正在发生吗？” - 你在没有验证的情况下做了假设
- “它会向我们显示……吗？” - 你本应添加证据收集
- “别猜了” - 你在未理解的情况下提出修复
- “Ultrathink this” - 质疑基本前提，而不只是症状
- “我们卡住了吗？”（沮丧） - 你的方法不起作用

**当你看到这些：** 停止。回到第 1 阶段。

## 常见合理化借口

| 借口 | 现实 |
|--------|---------|
| “问题很简单，不需要流程” | 简单问题也有根因。流程对简单 bug 也很快。 |
| “紧急情况，没时间走流程” | 系统化调试比猜测-检查式乱试更快。 |
| “先试这个，然后再调查” | 第一次修复会定下模式。一开始就做对。 |
| “确认修复有效后再写测试” | 未测试的修复站不住脚。先写测试才能证明它。 |
| “一次多个修复更省时间” | 无法隔离到底什么有效。还会造成新 bug。 |
| “参考太长，我会适配这个模式” | 部分理解保证会出 bug。完整阅读。 |
| “我看到问题了，让我修” | 看到症状 ≠ 理解根因。 |
| “再试一次修复”（2 次以上失败后） | 3 次以上失败 = 架构问题。质疑模式，不要继续修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. Root Cause** | 读取错误、复现、检查变更、收集证据 | 理解是什么以及为什么 |
| **2. Pattern** | 寻找可工作的示例、对比 | 识别差异 |
| **3. Hypothesis** | 形成理论、最小化测试 | 已确认或形成新假设 |
| **4. Implementation** | 创建测试、修复、验证 | Bug 已解决，测试通过 |

## 当流程揭示“没有根因”时

如果系统化调查显示问题确实是环境性、依赖时序或外部因素：

1. 你已经完成流程
2. 记录你调查了什么
3. 实现合适的处理（重试、超时、错误消息）
4. 添加监控/日志，便于未来调查

**但是：** 95% 的“没有根因”案例其实是调查不完整。

## 支持技术

这些技术属于系统化调试的一部分，并可在本目录中使用：

- **`root-cause-tracing.md`** - 沿调用栈反向追踪 bug，找到原始触发点
- **`defense-in-depth.md`** - 找到根因后，在多层添加验证
- **`condition-based-waiting.md`** - 用条件轮询替代任意超时

**相关技能：**
- **superpowers:test-driven-development** - 用于创建失败测试用例（第 4 阶段，第 1 步）
- **superpowers:verification-before-completion** - 在声称成功前验证修复确实有效

## 真实世界影响

来自调试会话：
- 系统化方法：15-30 分钟修好
- 随机修复方法：2-3 小时乱试
- 首次修复成功率：95% 对 40%
- 引入新 bug：几乎为零 对 常见
