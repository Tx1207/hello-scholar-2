# Landing / 落地

中文版本用于阅读和文案打磨。执行规则以英文 `SKILL.md` 为准。

## 概览

这个 skill 用在 `takeoff` 之后，或用户明确要求把一个已经打开的大胆方向压实的时候。`landing` 是 `takeoff` 的反面搭档：`takeoff` 负责把格局打开，`landing` 负责把大胆方向改写成可行方案。

只有存在前序大胆方向时才使用 `landing`。没有这个前提时，不要硬套落地结构，改按用户真实请求选择对应流程。

对 `landing` 来说，有效输入必须说清：bold thesis 是什么、它替代的旧模型是什么、主要现实疑问是什么。如果这些说不清，不要运行落地模板；先按普通问题回答，或先要求补齐方向。

它不是保守派，不是反重构，也不是第一步生成器。它的作用是把一个有杠杆但可能不现实的大方向，改成能承受真实契约、时间、风险和下游设计的版本。

## 核心原则

改方案，不是把方案砍成第一步。

可以有大想法，可以有彻底重构，可以有强架构判断。`landing` 要保留野心里有价值的部分，但改写经不起现实的部分。

一个有用的落地回答必须回答：

- 改写前，应该怎么给 takeoff 输出做价值排序？
- `takeoff` 里的哪些野心应该保留？
- takeoff thesis 里的哪些部分必须改写才真实？
- 哪些真实契约限制落地版方向？
- 落地版方案是什么，而不是只给第一步？
- 哪些阶段边界、验证和止损规则让方案负责任？

如果答案只是说第一步做什么，它还没有把方向落地，只是在绕开方案改写。

`landing` 和 `takeoff` 是一对：

- `takeoff` 问："如果不害怕，我们真正应该做什么？"
- `landing` 问："干净目标被现实改写后，应该变成什么可行目标？"

不要让 `landing` 抹掉大目标。要把方案改到真实可行，但保留它一开始值得打开格局的原因。

如果没有来自 `takeoff`、类似 takeoff 的架构讨论，或用户明确要求运行 `landing` 的前序大胆方向，不要因为"先验证"、"风险"、"MVP"、"止损"、"先做哪个"这类普通词汇推断应该使用本 skill。

## 工作流

1. 用一句话复述大方向。不要把野心磨平。说明它来自 `takeoff`、类似 takeoff 的架构审查，还是用户明确要求运行 `landing`。如果没有已经打开的大胆方向，或说不清 bold thesis / 旧模型 / 主要现实疑问，改用别的回答方式，不要硬套这个模板。

2. 先给 takeoff 输出做价值排序，再改写它。任务不是默认把 takeoff thesis 变小，也不是粗暴分成"有价值/没价值"两类。把主张、想法或机制分成四类：
   - **必须保留** —— 让 takeoff thesis 值得推进的核心价值；丢掉它，野心就塌了。
   - **改写后保留** —— 意图有价值，但形状、成本、迁移路径或契约匹配不现实。
   - **延后** —— 可能有价值，但当前落地版方向还不需要，或还没有证据/真实契约支撑。
   - **删除** —— 低价值、假价值、审美化复杂度、重复机制，或价值小于成本/风险。

   AI 的价值排序是基于证据的建议，不是最终裁决。如果用户不同意，不要把 AI 排序当权威硬顶，也不要无条件顺从；把用户判断当作新的约束，然后重新定价成本、风险、阶段边界、验证和止损规则。

3. 做现实检查。基于价值排序说清什么保留、什么必须改写、什么延后、什么删除。先扫描五种反模式，再点名它们暴露的约束：
   - **有愿景，没有可行形状** —— 目标很吸引人，但还不能被真实用户、代码、团队或测试操作。
   - **假迁移计划** —— 目标很干净，但路径默认所有东西可以一次改完。
   - **风险没有定价** —— 轻飘飘说"可以重构"，却没算数据丢失、影响面、测试缺口、隐藏调用方。
   - **野心塌缩成第一步** —— 回答给了一个很小的起手式，却没有把大想法改写成可工作的方案。
   - **没有止损规则** —— 计划只能继续，不能体面失败。

   然后回答：哪些真实契约限制了这件事？哪个区域影响面最大？哪些假设还没被证明？方案里哪些部分只是好看、猜测或过早？每种反模式的具体反制动作见 `references/anti-patterns.md`。

4. 产出落地版方向。这是 `landing` 的核心：把大胆方向改写成可行方案。定义约束定价后的目标模型、哪些保留、哪些改变、哪些延期，以及哪个下游阶段可以消费。

5. 定义阶段边界。说清落地版方案现在应该包含什么、什么留给设计阶段、什么留给实施计划、什么不该做。这不是完整执行计划，而是可行方向的边界。

6. 明确验证规则。成功标准必须可观察；失败信号必须提前命名；验证成本要低到在信心衰减前就能跑完。如果需要用测试驱动行为，就直接点明。

7. 定义止损规则。什么证据会推翻或暂停落地版方案？什么情况说明目标要缩小？什么东西可以回滚或隔离？哪个决策现在不要做？

8. 收尾时给出明确下一步：建议采用落地版方案、继续改写、暂停还是继续验证。询问用户是否按这个判断推进；如果下一步需要进入设计阶段，再询问是否进入 `brainstorming`。下一步必须询问用户一个明确问题，不能只陈述推荐阶段。不要自动切换阶段。

## 输出

产出简洁的落地判断。内容较复杂时使用这个顺序；短对话里保留同样的判断内容，但不要把回答写成文档：落地审判 / 价值排序 / 保留的野心 / 必须改写的部分 / 用户裁决点 / 现实检查 / 落地版方案 / 阶段边界 / 验证 / 止损规则 / 下一步。

守住这些约束：

- 开头先给判断结论——推进 / 缩小 / 暂停 / 拒绝 / 先验证——而不是先铺分析。
- 价值排序必须使用四类：必须保留、改写后保留、延后、删除。某一类为空可以明说没有；不要把价值压成"有价值/没价值"二分。
- 用户裁决点要说清用户判断可能在哪些地方推翻建议。如果发生这种情况，重新定价成本、风险、阶段边界、验证和止损规则。
- 把真实约束和焦虑惯性分开。
- 有价值时保留大目标，但要把它改写到真实可行。
- 默认是现实可行的落地版方案，不是只给第一步。除非那真的是唯一负责任的版本，否则不要把大胆方向降成一个象征性 MVP。
- 下一步必须询问用户是否推进、继续改写、暂停、继续验证，或在需要设计时是否进入 `brainstorming`。
- 不要写巨大的计划。`landing` 应该产出能被设计或实施计划消费的落地版方向，而不是完整执行结构。

## 它不是什么

- 不是通用项目管理。它是把 takeoff thesis 改写成可行方案的一次 rewrite pass。
- 不是反重构，也不是胆小。它反对的是幻想迁移，不是干净目标。
- 不是第一步生成器。第一步可以出现在验证或阶段边界里，但主输出是落地版方案。
