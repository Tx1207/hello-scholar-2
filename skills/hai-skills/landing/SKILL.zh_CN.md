# Landing / 落地

中文版本用于阅读和文案打磨。执行规则以英文 `SKILL.md` 为准。

## 概览

这个 skill 用在 `takeoff` 之后，或用户明确要求把一个已经打开的大胆方向压实的时候。`landing` 是 `takeoff` 的反面搭档：`takeoff` 负责把格局打开，`landing` 负责让已经选中的方向接触现实。

只有存在前序大胆方向时才使用 `landing`。没有这个前提时，不要硬套落地结构，改按用户真实请求选择对应流程。

它不是保守派，也不是反重构。它的作用是让一个大方向变得可执行、可验证、可止损，并且能在下一步产生真实进展。

## 核心原则

先把路踩实，再谈大胜利。

可以有大想法，可以有彻底重构，可以有强架构判断。但一个有用的方案必须回答：

- 最小证明动作是什么？
- 什么证据说明方向是对的？
- 哪些真实约束会把它打爆？
- 第一阶段应该主动砍掉什么？
- 如果判断错了，什么时候停？

如果一个方案说不清楚第一步怎么做，它还不是计划，只是一种情绪。

`landing` 和 `takeoff` 是一对：

- `takeoff` 问："如果不害怕，我们真正应该做什么？"
- `landing` 问："这个方向接触现实后的第一个证明点是什么？"

不要让 `landing` 抹掉大目标。它压缩的是第一步，不是野心。

如果没有来自 `takeoff`、类似 takeoff 的架构讨论，或用户明确要求运行 `landing` 的前序大胆方向，不要因为"先验证"、"风险"、"MVP"、"止损"、"先做哪个"这类普通词汇推断应该使用本 skill。

## 工作流

1. 用一句话复述大方向。不要把野心磨平。说明它来自 `takeoff`、类似 takeoff 的架构审查，还是用户明确要求运行 `landing`。如果没有已经打开的大胆方向，改用别的回答方式，不要硬套这个模板。

2. 做现实检查。先扫描五种反模式，再点名它们暴露的约束：
   - **有愿景，没有第一步** —— 听起来都对，但今天下午不知道该改哪个文件、写哪个测试。
   - **假迁移计划** —— 目标很干净，但路径默认所有东西可以一次改完。
   - **风险没有定价** —— 轻飘飘说"可以重构"，却没算数据丢失、影响面、测试缺口、隐藏调用方。
   - **长期正确，短期不负责任** —— 现在全量做会拖垮当前目标。
   - **没有止损规则** —— 计划只能继续，不能体面失败。

   然后回答：哪些真实契约限制了这件事？哪个区域影响面最大？哪些假设还没被证明？方案里哪些部分只是好看、猜测或过早？每种反模式的具体反制动作见 `references/anti-patterns.md`。

3. 选择最小可行推进。只选一个窄切片、证明点或决策产物。定义它改什么，也定义它拒绝改什么。优先选择能产生证据的动作，而不是继续写更多计划。

4. 明确验证规则。成功标准必须可观察；失败信号必须提前命名；验证成本要低到在信心衰减前就能跑完。如果需要用测试驱动行为，就直接点明。

5. 激进砍范围。列出第一步不应该尝试的东西。砍掉没有真实契约支撑的兼容工作、不影响证明点的架构打磨，以及在窄切片证明前的大迁移。

6. 定义止损规则。什么证据会推翻或暂停这个方向？什么情况说明目标要缩小？什么东西可以回滚或隔离？哪个决策现在不要做？

7. 收尾时给出明确下一步：建议采用、缩小、暂停还是继续验证。询问用户是否按这个判断推进；如果下一步需要进入设计阶段，再询问是否进入 `brainstorming`。不要自动切换阶段。

## 输出

产出简洁的落地判断。内容较复杂时使用这个顺序；短对话里保留同样的判断内容，但不要把回答写成文档：落地审判 / 保留的大方向 / 现实检查 / 最小可行推进 / 验证 / 砍掉清单 / 止损规则 / 下一步。

守住这些约束：

- 开头先给判断结论——推进 / 缩小 / 暂停 / 拒绝 / 先验证——而不是先铺分析。
- 把真实约束和焦虑惯性分开。
- 有价值时保留大目标，但不要让大目标代替执行。
- 默认是缩小验证，不是原地瘫痪。不要把 `landing` 变成"什么都别做"。
- 不要写巨大的计划。`landing` 应该产出被现实压力测试过的第一步，而不是完整执行结构。

## 它不是什么

- 不是通用项目管理。它专门审判方案能不能落地。
- 不是反重构，也不是胆小。它反对的是幻想迁移，不是干净目标——而且默认是缩小验证，不是原地不动。
