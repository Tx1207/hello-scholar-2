# Landing / 落地

中文版本用于阅读和文案打磨。执行规则以英文 `SKILL.md` 为准。

## 概览

自动触发只在 `takeoff`（包括 `geju`）已经产出方向、且该方向需要可行性压实时使用。它不是每次 `takeoff` 后的默认下一步；已认可的方向可以进 `brainstorming` 做设计。用户明确要求 `landing` 是有效触发，但仍然需要前序方向，或补齐 landing 所需输入。`landing` 负责把大胆方向改写成可行方案。

对 `landing` 来说，有效输入必须说清：bold thesis 是什么、它替代的旧模型是什么、主要现实疑问是什么。如果这些说不清，不要运行落地模板；先按普通问题回答，或先要求补齐方向。

核心原则：改方案，不是把方案砍成第一步。`landing` 要保留野心里有价值的部分，改写经不起现实的部分，并产出落地版方向。输出不是只给第一步，也不是完整执行计划。

如果没有前序方向，也没有用户明确要求 `landing`，不要因为"先验证"、"风险"、"MVP"、"架构"、"止损"、"先做哪个"这类普通词汇推断应该使用本 skill。

## 工作流

1. 用一句话复述大方向。不要把野心磨平。说明它来自前序 `takeoff`/`geju` 输出，还是用户明确要求 `landing`。如果没有前序方向，或说不清 bold thesis / 旧模型 / 主要现实疑问，停止套用模板，先要求补齐输入或按普通问题回答。

2. 先给 takeoff 输出做价值排序，再改写它。任务不是默认把 takeoff thesis 变小，也不是粗暴分成"有价值/没价值"两类。把主张、想法或机制分成四类：
   - **必须保留** —— 让 takeoff thesis 值得推进的核心价值；丢掉它，野心就塌了。
   - **改写后保留** —— 意图有价值，但形状、成本、迁移路径或契约匹配不现实。
   - **延后** —— 可能有价值，但当前落地版方向还不需要，或还没有证据/真实契约支撑。
   - **删除** —— 低价值、假价值、审美化复杂度、重复机制，或价值小于成本/风险。

   每个非平凡项都要说明：证据：支撑排序的具体代码、文档、测试、用户承诺或观察行为；为什么重要：它保护的具体野心或痛点；不处理的代价：会破坏、膨胀或继续含糊的东西；落地处理：保留、改写、延后或删除的规则。方向涉及真实文件或契约时，只写一行分类表不够。

   AI 的价值排序是基于证据的建议，不是最终裁决。如果用户不同意，不要把 AI 排序当权威硬顶，也不要无条件顺从；把用户判断当作新的约束，然后重新定价成本、风险、阶段边界、验证和止损规则。重新定价必须拆成五个独立维度：成本、风险、阶段边界、验证、止损。

3. 做现实检查。基于价值排序说清什么保留、什么必须改写、什么延后、什么删除。扫描五种反模式，再点名它们暴露的约束：
   - **有愿景，没有可行形状** —— 目标很吸引人，但还不能被真实用户、代码、团队或测试操作。
   - **假迁移计划** —— 目标很干净，但路径默认所有东西可以一次改完。
   - **风险没有定价** —— 轻飘飘说"可以重构"，却没算数据丢失、影响面、测试缺口、隐藏调用方。
   - **野心塌缩成第一步** —— 回答给了一个很小的起手式，却没有把大想法改写成可工作的方案。
   - **没有止损规则** —— 计划只能继续，不能体面失败。

   然后回答：哪些真实契约限制这件事，哪个区域影响面最大，哪些假设未证明，哪些只是好看/猜测/过早？具体反制动作见 `references/anti-patterns.md`。

4. 产出落地版方向：约束定价后的目标模型、哪些保留、哪些改变、哪些延期，以及哪个下游阶段可以消费。

5. 定义阶段边界：现在包含什么、什么留给设计阶段、什么留给实施计划、什么不该做。

6. 明确验证规则。成功标准必须可观察；失败信号必须提前命名；验证成本要低到在信心衰减前就能跑完。如果需要用测试驱动行为，就直接点明。

7. 定义止损规则。什么证据会推翻或暂停落地版方案？什么情况说明目标要缩小？什么东西可以回滚或隔离？哪个决策现在不要做？

8. 收尾时给出明确下一步：建议采用落地版方案、继续改写、暂停还是继续验证。询问用户是否按这个判断推进；如果下一步需要进入设计阶段，再询问是否进入 `brainstorming`。下一步必须询问用户一个明确问题，不能只陈述推荐阶段。不要自动切换阶段。

## 输出

产出简洁的落地判断。内容较复杂时使用这个顺序；短对话里保留同样的判断内容，但不要把回答写成文档：落地审判 / 价值排序 / 保留的野心 / 必须改写的部分 / 用户裁决点 / 现实检查 / 落地版方案 / 阶段边界 / 验证 / 止损规则 / 下一步。

守住这些约束：

- 开头先给判断结论——推进 / 缩小 / 暂停 / 拒绝 / 先验证——而不是先铺分析。
- 价值排序必须使用四类：必须保留、改写后保留、延后、删除。某一类为空可以明说没有；不要把价值压成"有价值/没价值"二分。
- 价值排序里的重要项必须给出证据、为什么重要、不处理的代价和落地处理；面对具体代码库落地时，不要只给浅表分类表。
- 用户裁决点要说清用户判断可能在哪些地方推翻建议。如果发生这种情况，重新定价成本、风险、阶段边界、验证和止损规则，并拆成五个独立维度：成本、风险、阶段边界、验证、止损。
- 把真实约束和焦虑惯性分开；有价值时保留大目标。
- 下一步必须询问用户是否推进、继续改写、暂停、继续验证，或在需要设计时是否进入 `brainstorming`。
