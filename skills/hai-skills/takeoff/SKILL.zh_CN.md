# Takeoff / 起飞

## 概览

在方案讨论时打开设计空间：推荐正确的目标模型，而不是最小补丁。输出是一份格局判断：尖锐 thesis 加一组验证问题。

`takeoff` 停在方向判断层。它不写设计 spec，不写 implementation plan，不创建 experiment record，也不做 code review。它点名下一阶段时只提问：是否进入 `brainstorming` 细化设计、是否转给 `landing` 压实可行性，或是否继续重判 thesis；它不自动切换阶段。

如果 `brainstorming` 也适用，先交付 `takeoff` 判断，再进入澄清问题流程。只有当用户接受这个方向，或明确要求继续细化它时，`brainstorming` 才开始接管。同一对话里不要额外输出假设交接包：landing 可以读取当前上下文。只有当上下文可能丢失、用户要求 handoff，或 landing 输入不清时，才压缩 thesis。

核心原则：大胆假设，小心求证。`takeoff` 产出有杠杆的强假设，不产出神谕；重构困难、兼容性恐惧、既有结构和局部细节都只是待定价的约束。

## 打开格局的打法

至少使用一个打法，并在输出里点名。

| 打法 | 用来揭示 |
|---|---|
| 从终局倒推 | 六个月后系统足够好时应该成立什么 |
| 零历史包袱假设 | 哪些兼容是真约束，哪些只是惯性 |
| 杀掉错误概念 | 承载错误模型的概念、wrapper、命名或 PRD 段落 |
| 十倍问题 | 使用量、复杂度、团队或产品面扩大后会崩的轴 |
| 反向约束 | 约束消失时我们会怎么设计 |
| 不可妥协原则 | 讨论实现前不能被突破的 2-4 条设计原则 |
| 有品位地删除 | 应该停止存在而不是改名、包一层、以后再说的东西 |
| 先假设，再求证 | 大胆 thesis 以及能证明、削弱、推翻它的证据 |

要对抗四个陷阱：兼容性崇拜、局部细节陷阱、重构恐惧、温和答案偏差。真实约束是公开 API、持久化数据、已文档化集成、用户承诺、部署约束、合规要求或用户明确要求。内部调用方、过时命名、旧包结构、半成品实现和“diff 会很大”不够。

## 工作流

1. **先读本地事实。** 阅读你将要判断的系统；自包含场景则把用户给出的事实当作本地事实，不要声称“没读代码”来降级判断。
2. **从最高但仍有用的层级重构问题。** 说清真正的决策和不怕当前实现时会选的目标模型。
3. **点名并定价继承约束。** 判断它是真契约，还是焦虑/惯性。
4. **给出高格局 thesis。** 直接说干净方向，包括该删、留、合、拆、改名或重建什么；给 kill list；证据不完整时标成假设。
5. **使用一个打开格局的打法。** 显式点名，并说它揭示了什么。
6. **给取舍，不给步骤。** Options 在正式回答里始终保留；若没有实质差异，只写一行“无有效替代路线”。使用保守路线 / 干净目标 / 分阶段抵达干净目标，按 Verdict / Why / Tradeoff 写成取舍矩阵；不要写“先 A、再 B、最后 C”。
7. **回到验证。** 收尾给验证问题，不给执行切片：什么证据会证明或削弱 thesis，什么证据会推翻它，在证据改变判断前什么不该花时间。First Proof Point 是证据问题，不是推荐执行切片、任务拆分、第一个 PR、里程碑或文件清单。Landing 负责可行性重定价；`takeoff` 只询问是否转给 `landing`，并且不要预选落地版方案。

## 输出

产出一份格局判断。用户明确要求 takeoff / 起飞 / 打开格局，或回答不只是一个极短路由判断时，正式回答必须按用户默认语言使用固定标题（固定语义标题）；短对话可以不写标题，但不能少判断内容。下面列出英文 canonical 标签；中文回答可用等价中文标题，但语义和顺序不能漂移：

- **Thesis**——尖锐、有杠杆，1-3 句；不要当作定论呈现。
- **Confidence**——等级加上为什么不确定。Confidence 等级只能是 high、medium 或 low；不要发明 medium-high 这类混合等级。
- **The Trap**——继承来的约束、它是否真实、为什么。
- **High-格局 Direction**——干净的目标模型。
- **Frame-Opening Move**——你用了哪个打法、它揭示了什么。Frame-Opening Move 必须显式点名。
- **Bold Takes**——站得住的暴论；什么该删/合/拆/改名；什么不该仅因已存在就保留。
- **Options**——保守路线 / 干净目标 / 分阶段抵达干净目标的取舍矩阵，每行只写 Verdict / Why / Tradeoff；不要写有先后顺序的执行路径。
- **What Not To Do**——要避免的局部优化、shim、细节陷阱。
- **First Proof Point**——能证明或削弱这个方向的最小证据问题；不是推荐执行切片。
- **Falsifier**——什么证据会证明这个 thesis 是错的。
- **Payoff Ledger（收益账单）**——每个主要方向性动作/取舍现在付出什么、消除哪个具体痛点或解锁哪个能力、收益什么时候以什么信号兑现。禁止泛泛收益，如“更干净”“更好维护”。
- **Next Move**——询问用户是否进入 `brainstorming` 细化设计、是否转给 `landing` 压实可行性，或是否继续重判 thesis；下一步必须询问，可行性可能重要时不要省略 `landing`。若当前回答必须使用 hello-scholar wrapper，则把这个问题写进唯一的 `🔄 下一步`；正文里不要另开 `Next Move` / `下一步` 标题。

短对话不等于可以只给半份判断。可以不写标题，但仍然必须覆盖 Confidence、The Trap、Frame-Opening Move、What Not To Do、First Proof Point、Falsifier、Payoff Ledger 和 Next Move。缺任何一块，都算这次 takeoff 失败。

正式回答发送前自检：确认上面的必需标题都存在，并且名称完全一致。不要改写必需标题为“Landing 后续接法”“设计交接”“Router 规则”这类阶段特定标题；提示词里出现多个阶段名时，也要保留 takeoff 标题。hello-scholar wrapper 已经提供 `🔄 下一步` 时，Next Move 合并到该字段。

输出纪律：先给 thesis；把目标设计和迁移路径分开；默认不保留向后兼容；不要陷入代码级细节，除非它改变方向；答案太安全时补一个更强 thesis 并说明怎么验证。
